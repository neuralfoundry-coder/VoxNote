## 1. 소개

### 1.1 목적

본 문서는 VoxNote 프로젝트의 소프트웨어 요구사항을 체계적으로 정의한다. 개발팀, QA팀, 프로젝트 관리자, 이해관계자가 시스템의 기능·비기능·인터페이스 요구사항을 명확히 이해하고, 다음 용도로 활용한다:

- **개발 기준:** 구현 범위와 우선순위 결정
- **테스트 기준:** 인수 조건 기반 테스트 케이스 도출
- **진행 추적:** 요구사항 ID 기반 개발 완료율 관리
- **변경 관리:** 요구사항 변경 시 영향도 분석의 기준

### 1.2 범위

VoxNote는 **로컬 AI 기반 차세대 회의록 서비스**로, 음성을 텍스트로 변환(STT)하고 AI를 활용하여 회의록을 자동 요약·문서화하는 크로스플랫폼 애플리케이션이다.

**포함 범위:**
- 실시간 오디오 캡처 및 전처리 파이프라인
- 로컬 AI 기반 STT, LLM, TTS, 화자 분리 엔진
- 외부 LLM Provider API 연동 (OpenAI, Anthropic, Google Gemini, Groq, Ollama 등)
- 전사 후처리 (오탈자 교정, 고유명사 매칭, 타임스탬프 정렬)
- 로컬 저장소 (E2EE 암호화, 전문검색)
- 디바이스 간 CRDT 동기화
- 크로스플랫폼 클라이언트 (macOS, Windows, Linux, iOS, Android, Web)
- 최소 서버 구성 (인증, 라이선스, 알림, 동기화 릴레이, 모델 CDN)
- 외부 서비스 연동 내보내기 (Notion, Slack, Confluence)

**제외 범위:**
- 서버 측 AI 추론 (VoxNote 자체 클라우드 AI 서비스 제공하지 않음)
- 영상 녹화 및 화면 공유 기능
- 실시간 협업 편집 (동기화는 비동기 CRDT 방식)

### 1.3 용어 정의

| 용어 | 정의 |
|------|------|
| STT | Speech-to-Text. 음성을 텍스트로 변환하는 기술 |
| LLM | Large Language Model. 대규모 언어 모델 |
| TTS | Text-to-Speech. 텍스트를 음성으로 변환하는 기술 |
| VAD | Voice Activity Detection. 음성 구간 감지. 무음 구간을 필터링하여 처리 효율을 높인다 |
| GGUF | GPT-Generated Unified Format. llama.cpp에서 사용하는 양자화 모델 파일 형식 |
| CRDT | Conflict-free Replicated Data Type. 분산 환경에서 충돌 없이 동기화되는 데이터 구조 |
| E2EE | End-to-End Encryption. 송신자와 수신자만 데이터를 복호화할 수 있는 암호화 방식 |
| RAG | Retrieval-Augmented Generation. 외부 데이터 검색 결과를 LLM 컨텍스트에 주입하여 답변 품질을 높이는 기법 |
| IPC | Inter-Process Communication. Tauri에서 프론트엔드(WebView)와 Rust 백엔드 간 통신 채널 |
| Provider | AI 모델 제공자. 로컬 모델(whisper.cpp, llama.cpp) 또는 외부 API(OpenAI, Anthropic 등) |
| 화자 분리 | Speaker Diarization. 오디오에서 발화자를 식별·구분하는 기술 |
| FTS | Full-Text Search. SQLite FTS5를 이용한 전문 검색 |
| 핫스왑 | Hot Swap. 앱 재시작 없이 모델이나 Provider를 런타임에 교체하는 기능 |
| 슬라이딩 윈도우 | Sliding Window. 겹침(overlap)을 두고 일정 크기의 오디오 청크를 순차 처리하는 방식 |
| GBNF | GGML BNF Grammar. llama.cpp에서 출력 형식(JSON, Markdown 등)을 강제하는 문법 규칙 |

## 2. 전체 설명

### 2.1 제품 개요

VoxNote는 기존 클라우드 의존형 회의록 서비스(예: 티로)의 한계를 극복하기 위해 설계된 Privacy-First 회의록 서비스이다. 핵심 AI 추론을 사용자 디바이스에서 로컬로 수행하되, 사용자 선택에 따라 외부 LLM Provider API도 자유롭게 활용할 수 있는 하이브리드 구조를 채택한다.

| 기존 서비스 한계 | VoxNote 해결 전략 |
|-----------------|------------------|
| 클라우드 의존 (AssemblyAI, OpenAI) | 100% 로컬 추론 기본 + 클라우드 API 선택적 사용 (opt-in) |
| 오프라인 시 기능 제한 | 완전한 오프라인 동작 — 로컬 모델 기준 네트워크 불필요 |
| 월 $7~29 구독 비용 | 일회성 구매 or 오픈소스 — 서버 비용 ≈ 0 |
| AI 학습 금지 약속 (신뢰 기반) | 기술적으로 불가능한 구조 — 로컬 모드 시 데이터 전송 자체 없음 |
| 웹/데스크탑/모바일 별도 개발 | 단일 Rust 코어 + Tauri 2.0 크로스플랫폼 |
| 제한된 언어 지원 (15개) | GGUF 양자화 모델 교체로 언어 무제한 확장 |
| 단일 AI Provider에 종속 | 로컬 모델 + 다수 외부 Provider 자유 선택 (trait 추상화) |

### 2.2 사용자 유형

| 사용자 유형 | 설명 | 주요 시나리오 |
|------------|------|-------------|
| **개인 사용자** | 개인 회의/강의 녹음 및 메모 용도 | 회의 녹음 → 전사 → 요약 → 검색. 로컬 모델 또는 클라우드 API 선택 사용 |
| **팀 사용자** | 팀 단위 회의록 공유 및 협업 | 팀 폴더에 회의록 저장, 공유 템플릿으로 문서 생성, CRDT 동기화로 멀티 디바이스 접근 |
| **관리자** | 팀 설정 및 구성원 관리 | 팀 템플릿 관리, 팀원 초대/권한 설정, 팀 단어장 관리 |

### 2.4 설계 제약사항

| 원칙 | 설명 | 근거 |
|------|------|------|
| LOCAL-FIRST | 모든 AI 추론은 기본적으로 디바이스 내에서 수행 | 프라이버시 보장, 오프라인 동작, 서버 비용 제거 |
| RUST-NATIVE | 코어 로직 100% Rust 구현 | 메모리 안전성, 고성능, 크로스플랫폼 단일 코드베이스 |
| ZERO-TRUST | 서버는 사용자 데이터에 접근 불가 | E2EE 기반. 서버 침해 시에도 데이터 안전 |
| PLUGIN-ARCH | 모델 및 Provider 핫스왑 가능 | 사용자가 런타임에 최적의 모델/Provider 선택 |
| CROSS-PLAT | 단일 코드베이스, 7개 타겟 | Tauri 2.0 + 공유 Rust 코어로 개발·유지보수 비용 최소화 |
| PRIVACY-HARD | 프라이버시를 기술적으로 보장 (로컬 모드 시) | 데이터 전송 경로 자체가 없는 아키텍처. 약속이 아닌 구조 |
| PROVIDER-AGNOSTIC | 로컬 모델과 외부 API를 동일한 추상화 계층에서 처리 | Provider 종속 방지, 사용자 선택권 극대화 |

### 2.5 가정 및 의존성

**가정:**
- 데스크탑 사용자는 최소 8GB RAM을 보유한다
- GPU 가속은 선택사항이며, CPU만으로도 모든 기능이 동작해야 한다
- 외부 LLM API 사용 시 사용자가 직접 API 키를 발급·관리한다
- 네트워크가 없는 환경에서도 로컬 모델 기반 핵심 기능은 100% 동작한다
- Tauri 2.0의 모바일 지원(iOS/Android)이 프로덕션 수준으로 안정적이다

**외부 의존성:**

| 의존성 | 용도 | 대안 |
|--------|------|------|
| whisper.cpp (whisper-rs) | STT 엔진 | 외부 STT API (OpenAI Whisper API 등) |
| llama.cpp | LLM 엔진 | 외부 LLM API (OpenAI, Anthropic 등) |
| Tauri 2.0 | 크로스플랫폼 셸 | Electron (성능 열위), Flutter (Rust FFI 복잡) |
| cpal | 오디오 I/O | 플랫폼 네이티브 API 직접 호출 |
| y-crdt (yrs) | CRDT 동기화 | Automerge (성능 비교 필요) |
| SQLite (rusqlite) | 로컬 DB | sled/redb (SQL 지원 부재) |
| ONNX Runtime (ort) | 화자 분리, TTS 모델 | libtorch (크기 과대) |

## 3. 기능 요구사항

> **우선순위 정의:**
> - **P0** — 필수. MVP에 반드시 포함. 미구현 시 릴리즈 불가
> - **P1** — 중요. 초기 릴리즈에 포함 목표. 일정에 따라 후순위 가능
> - **P2** — 개선. 후속 릴리즈에서 구현. 사용자 피드백에 따라 우선순위 조정

> **요구사항 항목 구조:**
> 각 요구사항은 ID, 요구사항 제목, 상세 설명, 인수 조건(AC), 의존성을 포함한다.

---

### 3.1 FR-AUD: 오디오 파이프라인

오디오 캡처부터 STT 입력까지의 전체 파이프라인을 관리한다. 마이크 입력 및 시스템 사운드를 캡처하고, 리샘플링·VAD 필터링을 거쳐 STT 엔진에 전달한다.

---

#### FR-AUD-001: 실시간 마이크 오디오 캡처

- **우선순위:** P0 | **Phase:** 1
- **설명:** cpal 크레이트를 사용하여 시스템의 기본 입력 디바이스(마이크)로부터 오디오를 실시간 캡처한다. 캡처 포맷은 48kHz, f32, 스테레오를 기본으로 하며, 사용자가 입력 디바이스를 선택할 수 있어야 한다.
- **인수 조건:**
  1. 앱 시작 후 사용 가능한 오디오 입력 디바이스 목록을 열거할 수 있다
  2. 선택한 디바이스에서 오디오 스트림을 시작/중지할 수 있다
  3. 캡처 중 디바이스 분리(disconnect) 시 에러 이벤트가 발생하고 UI에 표시된다
  4. macOS, Windows, Linux에서 동일하게 동작한다
- **의존성:** 없음

#### FR-AUD-002: 오디오 리샘플링

- **우선순위:** P0 | **Phase:** 1
- **설명:** rubato 크레이트를 사용하여 캡처된 오디오(48kHz, 스테레오)를 whisper.cpp 입력 요구사항인 16kHz mono f32 형식으로 변환한다. 리샘플링 품질은 sinc interpolation 수준을 유지한다.
- **인수 조건:**
  1. 입력 44.1kHz/48kHz/96kHz → 출력 16kHz 변환이 정상 동작한다
  2. 스테레오 → 모노 다운믹싱이 올바르게 수행된다
  3. 리샘플링에 의한 오디오 지연이 10ms 미만이다
  4. 리샘플링 아티팩트(aliasing, 클리핑)가 청취 가능 수준에서 발생하지 않는다
- **의존성:** FR-AUD-001

#### FR-AUD-003: 음성 구간 감지 (VAD)

- **우선순위:** P0 | **Phase:** 1
- **설명:** Silero VAD 또는 webrtc-vad를 사용하여 오디오 스트림에서 음성 구간만 필터링한다. 무음 또는 배경 소음 구간은 STT 엔진에 전달하지 않아 처리 효율을 높인다.
- **인수 조건:**
  1. 음성 시작 감지 후 50ms 이내에 음성 구간으로 전환된다
  2. 음성 종료 후 300ms~500ms 의 silence margin을 두고 비음성 구간으로 전환된다
  3. VAD 민감도(threshold)를 사용자가 설정 UI에서 조정할 수 있다 (0.0~1.0)
  4. 배경 음악, 키보드 소리 등 비음성 소음을 90% 이상 필터링한다
- **의존성:** FR-AUD-002

#### FR-AUD-004: Lock-free 링버퍼 기반 스레드 간 데이터 전달

- **우선순위:** P0 | **Phase:** 1
- **설명:** crossbeam-channel을 사용하여 오디오 캡처 스레드(실시간 콜백)와 STT 처리 스레드 간 lock-free 데이터 전달을 구현한다. 캡처 콜백의 실시간 제약을 위반하지 않아야 한다.
- **인수 조건:**
  1. 캡처 콜백 내에서 메모리 할당(malloc)이 발생하지 않는다
  2. 버퍼 오버플로우 시 가장 오래된 데이터를 드롭하고 경고 로그를 기록한다
  3. 1시간 연속 녹음에서 오디오 드롭 횟수가 5회 미만이다
- **의존성:** FR-AUD-001

#### FR-AUD-005: macOS 시스템 사운드 캡처

- **우선순위:** P1 | **Phase:** 4
- **설명:** ScreenCaptureKit(macOS 13+) 또는 CoreAudio Aggregate Device를 사용하여 시스템 오디오 출력(화상회의 소리 등)을 캡처한다. 마이크 입력과 시스템 사운드를 동시 캡처하여 별도 채널로 분리한다.
- **인수 조건:**
  1. Zoom, Google Meet, Teams 등 화상회의 앱의 오디오 출력을 캡처할 수 있다
  2. 마이크 입력과 시스템 사운드가 별도 채널로 분리되어 STT에 전달된다
  3. ScreenCaptureKit 권한 요청 다이얼로그가 적절히 표시된다
  4. macOS 13 미만 시 기능 비활성화 및 안내 메시지를 표시한다
- **의존성:** FR-AUD-001, FR-AUD-002

#### FR-AUD-006: Windows 시스템 사운드 캡처

  1. Windows 10/11에서 시스템 오디오 출력을 정상 캡처할 수 있다
  2. 마이크 입력과 시스템 사운드를 동시 캡처할 수 있다

#### FR-AUD-007: Linux 시스템 사운드 캡처

  3. 포그라운드 서비스로 녹음이 지속된다

### 3.2 FR-STT: 음성-텍스트 변환

whisper.cpp 기반의 로컬 STT 엔진과 외부 STT API를 통합하여, 오디오 입력을 텍스트로 변환한다. 실시간 스트리밍 전사와 파일 일괄 전사를 모두 지원한다.

---

#### FR-STT-001: 실시간 스트리밍 전사

- **우선순위:** P0 | **Phase:** 1
- **설명:** whisper-rs(whisper.cpp Rust 바인딩)를 사용하여 오디오 파이프라인에서 전달받은 오디오 청크를 실시간으로 전사한다. 전사는 별도 스레드에서 수행하며, 메인 스레드를 블로킹하지 않는다.
- **인수 조건:**
  1. 오디오 입력 시작 후 2초 이내에 첫 번째 전사 결과가 출력된다
  2. CPU 모드에서 실시간 배수(RTF) 1.0 이상을 유지한다 (base 모델 기준)
  3. GPU 모드에서 RTF 5.0 이상을 달성한다 (large-v3-turbo 기준, Apple M4)
  4. 전사 중 메모리 사용량이 모델 크기 + 500MB를 초과하지 않는다
- **의존성:** FR-AUD-003, FR-AUD-004

#### FR-STT-002: 슬라이딩 윈도우 기반 청크 처리

- **우선순위:** P0 | **Phase:** 1
- **설명:** 2~3초 단위의 슬라이딩 윈도우에 0.5초 오버랩을 적용하여 오디오 청크를 순차 처리한다. 오버랩 구간을 통해 단어 경계에서의 끊김을 방지한다.
- **인수 조건:**
  1. 윈도우 크기(2~3초)와 오버랩(0.3~1.0초)을 설정에서 조정할 수 있다
  2. 오버랩 구간의 중복 텍스트가 자동 제거(de-duplication)된다
  3. 연속된 윈도우 간 전사 텍스트가 자연스럽게 이어진다
- **의존성:** FR-STT-001

#### FR-STT-003: 문맥 연속성 유지

- **우선순위:** P0 | **Phase:** 1
- **설명:** 직전 전사 결과를 whisper의 initial_prompt 파라미터로 주입하여 연속된 발화의 문맥을 유지한다. 사용자 단어장의 고유명사도 prompt에 포함하여 인식률을 높인다.
- **인수 조건:**
  1. "오늘 회의 주제는"으로 끝난 직전 결과 이후 "프라이버시입니다"가 정상 연결된다
  2. 사용자 단어장에 등록된 고유명사(회사명, 제품명 등)의 인식률이 prompt 미주입 대비 30% 이상 향상된다
  3. initial_prompt 길이가 whisper 최대 토큰(224)을 초과하지 않도록 자동 truncation된다
- **의존성:** FR-STT-001, FR-STO-005

#### FR-STT-004: 다국어 자동 감지 및 수동 언어 선택

- **우선순위:** P0 | **Phase:** 1
- **설명:** whisper의 language detection 기능을 활용하여 발화 언어를 자동 감지한다. 사용자가 수동으로 언어를 고정할 수도 있다.
- **인수 조건:**
  1. "auto" 모드에서 한국어, 영어, 일본어, 중국어를 95% 이상 정확도로 감지한다
  2. 수동 언어 선택 시 해당 언어의 전사 정확도가 auto 대비 동등 이상이다
  3. 회의 중간에 언어를 변경할 수 있다 (설정 변경 → 다음 청크부터 적용)
  4. 지원 언어 목록: auto, ko, en, ja, zh, es, fr, de 및 whisper 지원 전체 언어
- **의존성:** FR-STT-001

#### FR-STT-005: GPU 가속 지원

- **우선순위:** P0 | **Phase:** 1
- **설명:** 플랫폼별 GPU 가속을 활용하여 STT 추론 성능을 극대화한다. GPU 미지원 환경에서는 자동으로 CPU 폴백한다.
- **인수 조건:**
  1. macOS: Metal 가속이 Apple M1+ 에서 정상 동작한다
  2. Windows/Linux: CUDA 가속이 GTX 1060+ 에서 정상 동작한다
  3. GPU 미감지 시 CPU 모드로 자동 전환되며 사용자에게 알림한다
  4. GPU/CPU 모드를 설정에서 수동 전환할 수 있다
- **의존성:** FR-STT-001

#### FR-STT-006: 다중 모델 지원

- **우선순위:** P0 | **Phase:** 1
- **설명:** whisper 모델 크기별(tiny~large-v3-turbo) 선택을 지원한다. 사용자 시스템 사양에 따라 적절한 모델을 추천하며, 런타임에 모델을 교체할 수 있다.
- **인수 조건:**
  1. 다음 모델이 지원된다: tiny(40MB), base(80MB), small(250MB), medium(800MB), large-v3-turbo(850MB)
  2. 각 모델의 크기, 필요 메모리, 예상 성능이 UI에 표시된다
  3. 모델 변경 시 현재 녹음 세션을 중단하지 않고 다음 청크부터 적용된다 (핫스왑)
- **의존성:** FR-MDL-001, FR-MDL-006

#### FR-STT-007: 실시간 번역 모드

- **우선순위:** P1 | **Phase:** 2
- **설명:** whisper의 translate 태스크를 활용하여 소스 언어 음성을 영어로 실시간 번역 전사한다. LLM을 활용한 추가 번역(영어 → 대상 언어)도 지원한다.
- **인수 조건:**
  1. whisper translate 모드로 비영어 음성 → 영어 텍스트 변환이 동작한다
  2. 원본 언어 전사와 번역 결과를 동시에 표시할 수 있다
  3. 번역 품질이 원본 전사 대비 의미 보존율 85% 이상이다
- **의존성:** FR-STT-001, FR-LLM-006

#### FR-STT-008: 실시간 전사 결과 스트리밍

- **우선순위:** P0 | **Phase:** 1
- **설명:** 전사 결과를 Tauri event emit(`stt:segment`)을 통해 프론트엔드로 실시간 전송한다. 각 세그먼트에는 텍스트, 타임스탬프, 신뢰도, 화자 ID가 포함된다.
- **인수 조건:**
  1. 세그먼트 이벤트 페이로드: `{ text, start_ms, end_ms, speaker_id?, confidence }`
  2. 이벤트 발행 간격이 슬라이딩 윈도우 주기(2~3초)와 일치한다
  3. 프론트엔드에서 이벤트 수신 후 16ms(1프레임) 이내에 UI 업데이트가 가능하다
- **의존성:** FR-STT-001

#### FR-STT-009: 파일 업로드 일괄 전사

- **우선순위:** P1 | **Phase:** 2
- **설명:** WAV, MP3, M4A, FLAC 등 오디오 파일을 업로드하여 일괄(batch) 전사한다. 실시간 전사보다 높은 정확도 설정(beam_size 증가 등)을 적용할 수 있다.
- **인수 조건:**
  1. WAV, MP3, M4A, FLAC, OGG 형식을 지원한다
  2. 1시간 분량 파일 전사 시 CPU < 8분, GPU < 2분 이내 완료된다 (large-v3-turbo)
  3. 전사 진행률을 0~100%로 표시한다
  4. 전사 중 취소가 가능하며, 취소 시 이미 전사된 부분은 보존한다
  5. 파일 크기 2GB까지 지원한다
- **의존성:** FR-STT-001, FR-AUD-002

#### FR-STT-010: 외부 STT API 연동

- **우선순위:** P1 | **Phase:** 2
- **설명:** OpenAI Whisper API, Google Cloud STT 등 외부 STT 서비스를 Provider로 사용할 수 있다. FR-API의 통합 추상화 인터페이스를 통해 로컬 STT와 동일한 방식으로 동작한다.
- **인수 조건:**
  1. OpenAI Whisper API를 통한 전사가 동작한다
  2. 로컬 STT와 동일한 세그먼트 형식(`stt:segment`)으로 결과가 반환된다
  3. API 호출 실패 시 로컬 모델로 자동 폴백한다 (FR-API-010 참조)
  4. 전송 전 사용자에게 오디오 데이터의 외부 전송을 고지한다
- **의존성:** FR-API-001, FR-API-008

### 3.3 FR-LLM: LLM 엔진

llama.cpp 기반 로컬 LLM과 외부 LLM API를 통합하여 회의록 요약, 템플릿 문서 생성, 질의응답(RAG), 번역, 오탈자 교정 등 텍스트 AI 기능을 제공한다.

---

#### FR-LLM-001: 로컬 LLM 추론

- **우선순위:** P0 | **Phase:** 2
- **설명:** llama.cpp(자체 FFI 또는 llama-cpp-2 크레이트)를 사용하여 GGUF 양자화 모델을 로드하고 로컬에서 텍스트 생성 추론을 수행한다.
- **인수 조건:**
  1. Qwen2.5-3B/7B/14B, Llama-3.1-8B, Gemma-2-9B 모델을 로드할 수 있다
  2. 컨텍스트 길이 8192~32768 토큰을 설정에서 조정할 수 있다
  3. GPU 오프로드 레이어 수(n_gpu_layers)를 조정할 수 있다
  4. temperature, top_p 등 생성 파라미터를 조정할 수 있다
  5. 모델 로드 시간이 7B 모델 기준 5초 이내이다 (mmap 활용)
- **의존성:** FR-MDL-001

#### FR-LLM-002: 실시간 회의 요약

- **우선순위:** P0 | **Phase:** 2
- **설명:** 1~2분 단위로 축적된 전사 텍스트를 구조화된 요약으로 변환한다. 이전 요약 컨텍스트를 포함하여 연속성을 유지하며, 결과를 스트리밍으로 UI에 표시한다.
- **인수 조건:**
  1. 요약 요청 시 10초 이내에 결과 생성이 시작된다 (7B 모델 기준)
  2. 요약에 핵심 논의 사항, 결정 사항, 액션 아이템이 포함된다
  3. 이전 요약과의 연속성이 유지된다 (컨텍스트 주입)
  4. 원문 대비 1/5~1/10 분량으로 압축된다
  5. 요약 중 녹음·전사가 중단되지 않는다
- **의존성:** FR-STT-001, FR-LLM-001

#### FR-LLM-003: 템플릿 기반 문서 생성

- **우선순위:** P0 | **Phase:** 2
- **설명:** 사용자 정의 템플릿(회의록 양식, 보고서 양식 등)과 전체 전사 텍스트를 입력으로 양식에 맞는 문서를 자동 생성한다.
- **인수 조건:**
  1. 기본 제공 템플릿: 회의록, 브레인스토밍 정리, 강의 노트, 1:1 면담 기록
  2. 사용자가 Markdown 형식으로 커스텀 템플릿을 생성/편집/삭제할 수 있다
  3. 템플릿에 플레이스홀더({{날짜}}, {{참석자}}, {{주요 논의}} 등)를 사용할 수 있다
  4. 생성된 문서가 템플릿 구조를 90% 이상 준수한다
- **의존성:** FR-LLM-001

#### FR-LLM-004: GBNF Grammar 기반 출력 형식 제어

- **우선순위:** P1 | **Phase:** 2
- **설명:** llama.cpp의 GBNF Grammar 기능을 사용하여 LLM 출력을 JSON, Markdown 등 특정 형식으로 강제한다.
- **인수 조건:**
  1. JSON 스키마를 지정하면 유효한 JSON만 출력된다
  2. Markdown 헤딩 구조를 지정하면 해당 구조를 따르는 Markdown이 출력된다
  3. Grammar 적용 시 생성 속도 저하가 20% 이내이다
- **의존성:** FR-LLM-001

#### FR-LLM-005: Ask VoxNote (RAG 질의응답)

- **우선순위:** P1 | **Phase:** 2
- **설명:** 전사 텍스트를 청크로 분할 → 임베딩 생성 → 벡터 저장 후, 사용자 질문에 대해 관련 청크를 검색하여 LLM 컨텍스트에 주입, 답변을 생성한다.
- **인수 조건:**
  1. "회의에서 예산에 대해 뭐라고 했지?" 같은 자연어 질문에 답변한다
  2. 답변에 출처 전사 텍스트의 타임스탬프가 포함되어 원본을 추적할 수 있다
  3. 단일 노트 또는 복수 노트를 대상으로 검색할 수 있다
  4. 임베딩 생성 시간이 1시간 분량 전사 기준 30초 이내이다
- **의존성:** FR-LLM-001, FR-STO-006

#### FR-LLM-006: 다국어 번역

- **우선순위:** P1 | **Phase:** 2
- **설명:** 전사 텍스트를 LLM을 활용하여 대상 언어로 번역한다. whisper translate(→영어)와 조합하여 비영어 → 비영어 번역도 가능하다.
- **인수 조건:**
  1. 한국어↔영어, 한국어↔일본어, 영어↔중국어 번역을 지원한다
  2. 번역 결과가 원문의 의미를 85% 이상 보존한다
  3. 전체 전사 텍스트 또는 선택 구간의 번역이 가능하다
- **의존성:** FR-LLM-001

#### FR-LLM-007: 스트리밍 토큰 출력

- **우선순위:** P0 | **Phase:** 2
- **설명:** LLM 텍스트 생성 시 토큰 단위로 결과를 스트리밍하여 UI에 점진적으로 표시한다. 사용자가 생성 중 취소할 수 있다.
- **인수 조건:**
  1. 첫 번째 토큰이 1초 이내에 UI에 표시된다 (7B 모델 기준)
  2. 토큰 생성 속도가 UI에 실시간 표시된다 (tokens/sec)
  3. 생성 중 "중지" 버튼으로 즉시 중단할 수 있다
  4. 중단 시 이미 생성된 텍스트는 보존된다
- **의존성:** FR-LLM-001

#### FR-LLM-008: GGUF 양자화 모델 지원

- **우선순위:** P0 | **Phase:** 2
- **설명:** Q4_K_M, Q5_0, Q5_K_M, Q8_0 등 다양한 양자화 수준의 GGUF 모델을 지원하여, 사용자 시스템 사양에 맞는 모델을 선택할 수 있다.
- **인수 조건:**
  1. Q4_K_M, Q5_0, Q5_K_M, Q8_0, F16 양자화를 지원한다
  2. 각 양자화 수준별 예상 RAM 사용량과 성능 차이를 UI에 표시한다
  3. 동일 모델의 다른 양자화 버전 간 전환이 가능하다
- **의존성:** FR-MDL-001

#### FR-LLM-009: LLM 세션 풀링

- **우선순위:** P1 | **Phase:** 2
- **설명:** LLM 세션(컨텍스트)을 재사용하여 반복 호출 시 모델 로드 및 컨텍스트 초기화 오버헤드를 제거한다.
- **인수 조건:**
  1. 동일 모델에 대한 연속 호출 시 두 번째 호출부터 응답 시작이 50% 이상 빨라진다
  2. 세션 풀 크기를 설정에서 조정할 수 있다 (기본: 2)
  3. 메모리 부족 시 가장 오래된 세션을 자동 해제한다
- **의존성:** FR-LLM-001

#### FR-LLM-010: 전사 오탈자 교정

- **우선순위:** P1 | **Phase:** 2
- **설명:** STT 결과의 오탈자, 문법 오류를 LLM 경량 패스로 교정한다. 원본을 보존하고 교정 제안을 별도로 표시한다.
- **인수 조건:**
  1. 교정 대상: 동음이의어 오류, 조사 오류, 명백한 인식 오류
  2. 교정 결과를 원본과 diff 형태로 표시한다
  3. 사용자가 교정 제안을 수락/거부할 수 있다
  4. 교정 처리 시간이 전사 텍스트 1분 분량 당 2초 이내이다
- **의존성:** FR-LLM-001, FR-STT-001

### 3.4 FR-API: 외부 LLM Provider API 연동

로컬 모델 외에 주요 클라우드 LLM Provider API를 설정 기반으로 자유롭게 사용할 수 있도록 한다. 통합 추상화 인터페이스를 통해 로컬과 클라우드를 동일하게 취급한다.

---

#### FR-API-001: Provider 설정 관리 UI

- **우선순위:** P0 | **Phase:** 2
- **설명:** 설정 화면 내 AI Provider 패널에서 각 Provider의 활성화 여부, API 키, 엔드포인트, 기본 모델을 관리한다.
- **인수 조건:**
  1. Provider 목록(Local, OpenAI, Anthropic, Google Gemini, Groq, Ollama, Custom)이 표시된다
  2. 각 Provider별 API 키 입력 필드가 제공되며, 입력값은 마스킹(***) 처리된다
  3. API 키 입력 후 "연결 테스트" 버튼으로 유효성을 검증할 수 있다
  4. Provider별 사용 가능한 모델 목록을 API 호출로 자동 조회한다
  5. 설정 변경이 즉시 반영된다 (앱 재시작 불필요)
- **의존성:** 없음

#### FR-API-002: OpenAI API 연동

- **우선순위:** P0 | **Phase:** 2
- **설명:** OpenAI의 Chat Completions API, Whisper API, TTS API를 지원한다. GPT-4o, GPT-4o-mini 등 모델 선택이 가능하다.
- **인수 조건:**
  1. Chat Completions API (v1/chat/completions)를 통한 LLM 기능이 동작한다
  2. Whisper API (v1/audio/transcriptions)를 통한 STT 기능이 동작한다
  3. TTS API (v1/audio/speech)를 통한 음성 합성이 동작한다
  4. 스트리밍 응답(SSE)을 지원한다
  5. 사용 가능한 모델 목록을 v1/models API로 자동 조회한다
- **의존성:** FR-API-001, FR-API-008

#### FR-API-003: Anthropic API 연동

- **우선순위:** P0 | **Phase:** 2
- **설명:** Anthropic Messages API를 통해 Claude 모델을 LLM Provider로 사용한다.
- **인수 조건:**
  1. Messages API (v1/messages)를 통한 LLM 기능이 동작한다
  2. 스트리밍 응답(SSE)을 지원한다
  3. Claude Opus, Sonnet, Haiku 모델을 선택할 수 있다
  4. system prompt, temperature 등 파라미터를 전달할 수 있다
- **의존성:** FR-API-001, FR-API-008

#### FR-API-004: Google Gemini API 연동

- **우선순위:** P1 | **Phase:** 2
- **설명:** Google Gemini API를 통해 Gemini 모델을 LLM Provider로 사용한다.
- **인수 조건:**
  1. Gemini generateContent API를 통한 LLM 기능이 동작한다
  2. 스트리밍 응답을 지원한다
  3. Gemini Pro, Gemini Flash 모델을 선택할 수 있다
- **의존성:** FR-API-001, FR-API-008

#### FR-API-005: Groq API 연동

- **우선순위:** P2 | **Phase:** 3
- **설명:** Groq의 고속 추론 API(OpenAI-compatible)를 LLM Provider로 사용한다.
- **인수 조건:**
  1. OpenAI-compatible 엔드포인트를 통한 LLM 기능이 동작한다
  2. Groq 지원 모델(Llama, Mixtral 등)을 선택할 수 있다
  3. 응답 속도가 다른 클라우드 Provider 대비 유의미하게 빠르다
- **의존성:** FR-API-001, FR-API-008, FR-API-011

#### FR-API-006: Ollama 연동

- **우선순위:** P1 | **Phase:** 2
- **설명:** 사용자가 자체 운영하는 Ollama 서버를 OpenAI-compatible 엔드포인트로 연동한다. 로컬 네트워크 내 다른 PC의 Ollama도 지원한다.
- **인수 조건:**
  1. Ollama의 OpenAI-compatible API를 통한 LLM 기능이 동작한다
  2. 기본 엔드포인트 `http://localhost:11434`가 프리셋으로 제공된다
  3. Ollama에 설치된 모델 목록을 자동 조회할 수 있다
  4. 로컬 네트워크의 다른 IP:PORT 설정이 가능하다
- **의존성:** FR-API-001, FR-API-008, FR-API-011

#### FR-API-007: 엔진별 독립 Provider 선택

- **우선순위:** P0 | **Phase:** 2
- **설명:** STT, LLM, TTS 각 엔진에 대해 독립적으로 Provider를 선택할 수 있다. 예: STT는 로컬 whisper, LLM은 Anthropic Claude, TTS는 OpenAI TTS.
- **인수 조건:**
  1. STT Provider 선택: Local(whisper.cpp) / OpenAI Whisper API / Google STT
  2. LLM Provider 선택: Local(llama.cpp) / OpenAI / Anthropic / Gemini / Groq / Ollama / Custom
  3. TTS Provider 선택: Local(Piper) / OpenAI TTS / Google TTS
  4. 각 엔진의 Provider를 독립적으로 설정할 수 있다
  5. Provider 변경 시 현재 진행 중인 작업에 영향을 주지 않는다 (다음 요청부터 적용)
- **의존성:** FR-API-001

#### FR-API-008: 통합 추상화 인터페이스 (Provider Trait)

- **우선순위:** P0 | **Phase:** 2
- **설명:** 로컬 모델과 클라우드 API를 동일한 Rust trait으로 추상화하여, 호출 측 코드가 Provider 종류를 알 필요 없이 동일한 인터페이스를 사용한다.
- **인수 조건:**
  1. `SttProvider` trait: `transcribe(audio) -> Result<Vec<Segment>>` — 로컬/클라우드 구현이 존재한다
  2. `LlmProvider` trait: `generate(prompt, config) -> Result<Stream<Token>>` — 로컬/클라우드 구현이 존재한다
  3. `TtsProvider` trait: `synthesize(text) -> Result<AudioData>` — 로컬/클라우드 구현이 존재한다
  4. Provider 교체 시 호출 측 코드 변경이 불필요하다
  5. 런타임에 Provider를 동적으로 교체할 수 있다 (핫스왑)
- **의존성:** 없음

#### FR-API-009: API 키 보안 저장

- **우선순위:** P0 | **Phase:** 2
- **설명:** 외부 Provider의 API 키를 OS 네이티브 키체인에 암호화 보관한다. 평문 저장(파일, DB, 환경변수)을 금지한다.
- **인수 조건:**
  1. macOS: Keychain Services API를 사용한다
  2. Windows: Windows Credential Manager(DPAPI)를 사용한다
  3. Linux: Secret Service API(libsecret / GNOME Keyring)를 사용한다
  4. 키체인 미지원 환경에서는 age 암호화된 로컬 파일에 저장하고 사용자에게 고지한다
  5. 앱 메모리 덤프에서 API 키가 평문으로 노출되지 않는다 (zeroize on drop)
- **의존성:** 없음

#### FR-API-010: 네트워크 장애 시 로컬 폴백

- **우선순위:** P1 | **Phase:** 2
- **설명:** 외부 API 호출 실패(네트워크 단절, 타임아웃, 429/5xx 에러) 시 자동으로 로컬 모델로 폴백한다. 폴백 시 사용자에게 알림한다.
- **인수 조건:**
  1. API 호출 타임아웃: 10초 (설정 가능)
  2. 연속 3회 실패 시 해당 세션의 나머지 요청을 로컬 모델로 자동 전환한다
  3. 폴백 발생 시 UI 토스트 알림: "네트워크 불안정. 로컬 모델로 전환되었습니다"
  4. 네트워크 복구 감지 시 사용자 확인 후 외부 API로 복귀할 수 있다
  5. 로컬 모델이 다운로드되지 않은 경우 에러 메시지와 모델 다운로드 안내를 표시한다
- **의존성:** FR-API-008, FR-LLM-001

#### FR-API-011: 사용자 정의 엔드포인트

- **우선순위:** P1 | **Phase:** 2
- **설명:** OpenAI-compatible API를 제공하는 임의의 서비스(LM Studio, vLLM, text-generation-webui 등)에 연결할 수 있도록 사용자 정의 엔드포인트 URL을 설정한다.
- **인수 조건:**
  1. Base URL, API 키, 모델 이름을 수동 입력할 수 있다
  2. 연결 테스트로 엔드포인트 유효성을 검증할 수 있다
  3. 복수의 커스텀 엔드포인트를 등록할 수 있다 (이름 지정)
  4. OpenAI Chat Completions 호환 API 형식을 따른다
- **의존성:** FR-API-001, FR-API-008

#### FR-API-012: API 사용량 모니터링

- **우선순위:** P2 | **Phase:** 3
- **설명:** 외부 API 호출 시 사용된 토큰 수, 예상 비용, 호출 횟수를 추적하고 대시보드에 표시한다.
- **인수 조건:**
  1. Provider별 일간/월간 토큰 사용량을 표시한다
  2. Provider별 요금 테이블을 기반으로 예상 비용을 계산한다
  3. 월간 예산 한도를 설정하고, 도달 시 경고 알림을 표시한다
  4. 사용량 데이터를 CSV로 내보낼 수 있다
- **의존성:** FR-API-001

#### FR-API-013: Provider 플러그인 구조

- **우선순위:** P0 | **Phase:** 2
- **설명:** 새로운 Provider를 추가할 때 trait 구현체만 작성하면 되도록 플러그인 구조를 설계한다. Provider 등록은 레지스트리 패턴으로 관리한다.
- **인수 조건:**
  1. 새 LLM Provider 추가 시 `LlmProvider` trait 구현 + 레지스트리 등록만으로 완료된다
  2. 기존 코드 변경 없이 Provider가 설정 UI에 자동 노출된다
  3. Provider별 고유 설정(엔드포인트 형식, 인증 방식 등)을 메타데이터로 정의할 수 있다
- **의존성:** FR-API-008

### 3.5 FR-POST: 전사 후처리 파이프라인

STT 결과를 보정·보강하는 후처리 파이프라인이다. 고유명사 매칭, 화자 태깅, 타임스탬프 정렬을 포함한다.

---

#### FR-POST-001: 고유명사 자동 매칭

- **우선순위:** P1 | **Phase:** 2
- **설명:** 사용자 단어장에 등록된 고유명사(회사명, 제품명, 인명 등)를 Aho-Corasick 알고리즘으로 전사 텍스트에서 매칭·교체한다.
- **인수 조건:**
  1. 1,000개 이상의 단어장 항목에서도 실시간 매칭이 가능하다 (청크 당 1ms 이내)
  2. 매칭 시 원본과 교체어를 모두 기록한다 (되돌리기 가능)
  3. 대소문자 무시, 유니코드 정규화 옵션을 제공한다
- **의존성:** FR-STO-005

#### FR-POST-002: 타임스탬프 정렬

- **우선순위:** P0 | **Phase:** 1
- **설명:** whisper 출력의 세그먼트별 타임스탬프(start_ms, end_ms)를 오디오 원본과 정렬하여, 텍스트 클릭 시 해당 시점의 오디오를 재생할 수 있도록 한다.
- **인수 조건:**
  1. 타임스탬프 정확도: 실제 발화 시점 대비 ±200ms 이내
  2. 전사 텍스트의 임의 위치 클릭 시 해당 타임스탬프의 오디오로 이동할 수 있다
  3. 슬라이딩 윈도우 오버랩에 의한 타임스탬프 중복이 자동 해결된다
- **의존성:** FR-STT-001, FR-STT-002

#### FR-POST-003: 화자 태깅 통합

- **우선순위:** P1 | **Phase:** 2
- **설명:** 화자 분리(FR-DIA) 결과를 전사 세그먼트에 통합하여 각 텍스트에 화자 ID를 태깅한다.
- **인수 조건:**
  1. 각 전사 세그먼트에 speaker_id 필드가 할당된다
  2. 화자 교체가 정확히 반영된다 (화자 변경 시점의 정확도 ±500ms)
  3. 화자 ID에 사용자가 할당한 이름이 표시된다
- **의존성:** FR-DIA-001, FR-DIA-004

### 3.6 FR-TTS: 음성 합성

회의 내용 음성 재생 및 접근성 지원을 위한 TTS 엔진이다.

---

#### FR-TTS-001: 로컬 음성 합성 (Piper TTS)

- **우선순위:** P2 | **Phase:** 3
- **설명:** Piper TTS(ONNX 기반, ~20MB 모델)를 사용하여 텍스트를 로컬에서 음성으로 합성한다.
- **인수 조건:**
  1. 한국어, 영어, 일본어 음성 모델이 제공된다
  2. CPU에서도 실시간 합성이 가능하다 (RTF > 1.0)
  3. 음성 속도 조절이 가능하다 (0.5x ~ 2.0x)
  4. 합성된 음성을 WAV 파일로 저장할 수 있다
- **의존성:** FR-MDL-001

#### FR-TTS-002: 외부 TTS API 연동

- **인수 조건:**
  1. OpenAI TTS API를 통한 음성 합성이 동작한다
  2. 로컬 TTS와 동일한 인터페이스(TtsProvider trait)로 사용된다
  3. 음성 모델/스타일 선택이 가능하다
- **의존성:** FR-API-001, FR-API-008

### 3.7 FR-DIA: 화자 분리

오디오에서 발화자를 식별하고 전사 텍스트에 화자 정보를 태깅한다.

---

#### FR-DIA-001: 화자 임베딩 추출

- **우선순위:** P1 | **Phase:** 2
- **설명:** ECAPA-TDNN(ONNX Runtime 기반, ~20MB) 모델을 사용하여 VAD로 감지된 음성 구간마다 화자 임베딩 벡터를 추출한다.
- **인수 조건:**
  1. 각 음성 구간(0.5~3초)에서 고정 차원(192 또는 256)의 임베딩을 추출한다
  2. ONNX Runtime CPU 추론으로 실시간 처리가 가능하다 (음성 구간 당 < 50ms)
  3. 동일 화자의 임베딩 간 코사인 유사도가 0.75 이상이다
- **의존성:** FR-AUD-003

#### FR-DIA-002: 실시간 화자 클러스터링

- **우선순위:** P1 | **Phase:** 2
- **설명:** 온라인 Agglomerative 클러스터링으로 새 임베딩을 기존 클러스터와 비교하여 화자를 구분한다.
- **인수 조건:**
  1. 새 임베딩이 기존 클러스터와 코사인 유사도 임계값 이상이면 해당 화자로 분류한다
  2. 임계값 미만이면 새 화자로 분류한다
  3. 2~10명의 화자를 정확히 구분한다 (4명 기준 정확도 85% 이상)
  4. 실시간으로 동작하며 전사 지연에 영향을 주지 않는다
- **의존성:** FR-DIA-001

#### FR-DIA-003: 화자 분리 민감도 조정

- **우선순위:** P2 | **Phase:** 2
- **설명:** 코사인 유사도 임계값(0.65~0.75)을 사용자가 설정에서 조정할 수 있다.
- **인수 조건:**
  1. 임계값을 0.50~0.90 범위에서 슬라이더로 조정할 수 있다
  2. 낮은 값: 더 많은 화자로 분리 (세밀), 높은 값: 적은 화자로 병합 (보수적)
  3. 설정 변경 시 기존 분리 결과를 재계산할 수 있다 (선택적)
- **의존성:** FR-DIA-002

#### FR-DIA-004: 전사 세그먼트 화자 태깅

- **우선순위:** P1 | **Phase:** 2
- **설명:** 화자 클러스터링 결과를 전사 세그먼트의 speaker_id 필드에 기록한다.
- **인수 조건:**
  1. 각 전사 세그먼트에 speaker_id가 할당된다
  2. speaker_id는 "Speaker 1", "Speaker 2" 형식의 자동 라벨이다
  3. 화자 교체 시점이 전사 세그먼트 경계와 정렬된다
- **의존성:** FR-DIA-002, FR-STT-001

#### FR-DIA-005: 화자 이름 할당

- **우선순위:** P1 | **Phase:** 2
- **설명:** 사용자가 자동 라벨("Speaker 1")에 실제 이름을 할당할 수 있다. 이후 녹음에서도 임베딩 매칭으로 동일 화자를 자동 인식한다.
- **인수 조건:**
  1. UI에서 화자 라벨을 클릭하여 이름을 편집할 수 있다
  2. 이름 변경 시 해당 화자의 모든 세그먼트에 일괄 반영된다
  3. 화자 임베딩 프로필을 저장하여 다음 녹음에서 재사용할 수 있다 (선택적)
- **의존성:** FR-DIA-004

### 3.8 FR-STO: 로컬 저장소

모든 사용자 데이터를 디바이스 로컬에 저장하고, E2EE 암호화 및 전문검색을 지원한다.

---

#### FR-STO-001: SQLite 기반 데이터 저장

- **우선순위:** P0 | **Phase:** 1
- **설명:** rusqlite(FTS5 기능 포함)를 사용하여 노트, 전사 텍스트, 요약, 설정 등 모든 구조화 데이터를 SQLite에 저장한다.
- **인수 조건:**
  1. 노트 1,000건, 전사 세그먼트 100만 건 규모에서 조회 응답이 100ms 이내이다
  2. WAL(Write-Ahead Logging) 모드로 읽기/쓰기 동시성을 보장한다
  3. DB 파일 손상 시 자동 감지 및 복구 시도를 수행한다
  4. 마이그레이션 시스템으로 스키마 버전을 관리한다
- **의존성:** 없음

#### FR-STO-002: 전문검색 (FTS5)

- **우선순위:** P0 | **Phase:** 1
- **설명:** SQLite FTS5를 활용하여 전사 텍스트에 대한 전문검색을 지원한다. 한국어/영어/일본어 토큰화를 처리한다.
- **인수 조건:**
  1. 키워드 검색 시 100만 세그먼트에서 200ms 이내에 결과를 반환한다
  2. 한국어 형태소 분석 기반 토큰화를 지원한다 (또는 unicode61 토크나이저)
  3. 검색 결과에 매칭 하이라이트를 표시한다
  4. 전체 노트 또는 특정 노트/폴더 범위를 지정하여 검색할 수 있다
- **의존성:** FR-STO-001

#### FR-STO-003: KV Store 캐시

- **우선순위:** P1 | **Phase:** 1
- **설명:** redb를 사용하여 오디오 청크 캐시, 세션 상태, 임시 데이터를 고성능으로 저장한다.
- **인수 조건:**
  1. 오디오 청크 임시 저장 시 쓰기 지연이 1ms 이내이다
  2. 세션 상태(녹음 중/일시정지/재개) 복구가 가능하다
  3. TTL 기반 자동 만료로 캐시 크기를 관리한다
- **의존성:** 없음

#### FR-STO-004: E2EE 데이터 암호화

- **우선순위:** P0 | **Phase:** 1
- **설명:** age(X25519) 기반으로 저장 데이터를 암호화한다. 사용자 비밀번호에서 Argon2로 키를 파생하며, 데이터 파일은 항상 암호화 상태를 유지한다.
- **인수 조건:**
  1. DB 파일을 직접 열어도 평문 데이터를 확인할 수 없다
  2. 사용자 비밀번호 변경 시 마스터 키를 재암호화한다 (데이터 재암호화 불필요)
  3. 암호화/복호화 오버헤드가 10% 이내이다
  4. 암호화 키 분실 시 데이터 복구가 불가능하다 (의도된 동작)
- **의존성:** 없음

#### FR-STO-005: 사용자 단어장 관리

- **우선순위:** P1 | **Phase:** 2
- **설명:** 고유명사, 전문 용어, 약어 등을 사용자 단어장으로 관리한다. STT initial_prompt 주입 및 후처리 교정에 활용된다.
- **인수 조건:**
  1. 단어, 교체어, 도메인(업무/의료/법률 등) 필드로 관리한다
  2. CSV 가져오기/내보내기를 지원한다
  3. 도메인별 활성화/비활성화가 가능하다 (회의 성격에 따라 전환)
  4. 단어장 항목 수에 실질적 제한이 없다 (10,000건 이상 지원)
- **의존성:** FR-STO-001

#### FR-STO-006: 임베딩 벡터 저장 (RAG)

- **우선순위:** P1 | **Phase:** 2
- **설명:** 전사 텍스트 청크의 임베딩 벡터를 SQLite BLOB으로 저장하고, 코사인 유사도 기반 검색을 수행한다.
- **인수 조건:**
  1. 임베딩 차원: 384~768 (모델에 따라 가변)
  2. 10,000개 벡터에서 Top-K 검색이 100ms 이내에 완료된다
  3. 새 전사 완료 시 임베딩이 자동 생성되어 저장된다 (백그라운드)
  4. 사용하지 않는 임베딩의 자동 정리(garbage collection)를 지원한다
- **의존성:** FR-STO-001, FR-LLM-001

#### FR-STO-007: 폴더 구조 관리

- **우선순위:** P0 | **Phase:** 1
- **설명:** 노트를 폴더/하위 폴더 구조로 정리한다. 드래그 앤 드롭으로 이동 가능하다.
- **인수 조건:**
  1. 폴더 생성, 이름 변경, 삭제가 가능하다
  2. 최대 5단계 중첩 폴더를 지원한다
  3. 노트를 폴더 간 드래그 앤 드롭으로 이동할 수 있다
  4. 폴더 삭제 시 하위 노트 처리 옵션: 함께 삭제 / 상위 폴더로 이동
- **의존성:** FR-STO-001

### 3.9 FR-SYN: 디바이스 간 동기화

CRDT 기반으로 디바이스 간 노트를 동기화한다. 모든 동기화 데이터는 E2EE로 암호화되며, 서버는 평문에 접근할 수 없다.

---

#### FR-SYN-001: CRDT 기반 노트 동기화

- **우선순위:** P1 | **Phase:** 3
- **설명:** y-crdt(yrs)를 사용하여 노트 변경사항을 CRDT 문서로 관리하고, 디바이스 간 동기화한다. 충돌은 CRDT 특성상 자동 해결된다.
- **인수 조건:**
  1. 두 디바이스에서 동시에 같은 노트를 편집해도 데이터 유실이 없다
  2. 동기화 후 모든 디바이스에서 동일한 최종 상태가 보장된다
  3. 동기화 대상: 노트 텍스트, 요약, 폴더 구조, 설정
  4. 오디오 원본은 동기화 대상에서 제외한다 (용량 문제)
- **의존성:** FR-STO-001

#### FR-SYN-002: 동기화 데이터 E2EE 암호화

- **우선순위:** P1 | **Phase:** 3
- **설명:** CRDT 델타를 서버로 전송하기 전 age(X25519)로 암호화한다. 서버는 암호화된 바이너리만 중계하며, 복호화 키를 보유하지 않는다.
- **인수 조건:**
  1. 서버 DB를 직접 조회해도 동기화 데이터의 평문을 확인할 수 없다
  2. 새 디바이스 추가 시 기존 디바이스에서 키 교환이 필요하다
  3. 암호화 키 교환은 QR 코드 또는 수동 코드 입력으로 수행한다
- **의존성:** FR-SYN-001, FR-STO-004

#### FR-SYN-003: 서버 릴레이 무접근성 보장

- **우선순위:** P1 | **Phase:** 3
- **설명:** 동기화 릴레이 서버는 암호화된 바이너리 델타를 수신하여 대상 디바이스들에게 그대로 전달한다. 서버 측에서 복호화를 시도할 수 없는 구조를 보장한다.
- **인수 조건:**
  1. 서버 코드에 복호화 로직이 존재하지 않는다 (코드 리뷰로 검증)
  2. 서버가 저장하는 데이터는 암호화된 바이너리와 메타데이터(디바이스 ID, 타임스탬프)뿐이다
  3. 보안 감사에서 서버 측 데이터 접근 불가를 증명할 수 있다
- **의존성:** FR-SYN-002

#### FR-SYN-004: 오프라인 디바이스 버퍼링

- **우선순위:** P1 | **Phase:** 3
- **설명:** 수신 디바이스가 오프라인인 경우 서버에서 암호화된 델타를 임시 버퍼링한다. TTL은 30일이며, 만료 시 자동 삭제한다.
- **인수 조건:**
  1. 오프라인 디바이스가 30일 이내에 접속하면 누적 델타가 전달된다
  2. TTL 만료 시 버퍼링 데이터가 자동 삭제되며, 접속 시 전체 동기화를 수행한다
  3. 디바이스당 버퍼 용량 상한: 100MB
- **의존성:** FR-SYN-001

#### FR-SYN-005: WebSocket 실시간 동기화

- **우선순위:** P1 | **Phase:** 3
- **설명:** tokio-tungstenite를 사용하여 WebSocket 연결로 실시간 양방향 동기화를 수행한다.
- **인수 조건:**
  1. 연결 수립 후 1초 이내에 대기 중인 델타가 전송된다
  2. 연결 끊김 시 자동 재연결을 시도한다 (exponential backoff, 최대 5분)
  3. 하트비트(30초 간격)로 연결 상태를 유지한다
  4. 동시 접속 디바이스 수: 계정당 최대 10대
- **의존성:** FR-SYN-001

### 3.10 FR-MDL: 모델 관리

AI 모델의 다운로드, 검증, 교체, 추천을 관리한다.

---

#### FR-MDL-001: 모델 카탈로그 조회

- **우선순위:** P0 | **Phase:** 1
- **설명:** registry.toml 기반 모델 카탈로그를 제공한다. 로컬 캐시와 CDN 최신 버전을 동기화하여 사용 가능한 모델 목록을 표시한다.
- **인수 조건:**
  1. 각 모델의 ID, 이름, 타입(stt/llm/tts/diarization), 크기, 양자화, 지원 언어, 최소 RAM이 표시된다
  2. 이미 다운로드된 모델과 미다운로드 모델을 구분하여 표시한다
  3. CDN에서 최신 카탈로그를 주기적으로(앱 시작 시 + 24시간 간격) 동기화한다
  4. 오프라인 시 로컬 캐시된 카탈로그를 사용한다
- **의존성:** 없음

#### FR-MDL-002: 모델 다운로드 (재개 가능)

- **우선순위:** P0 | **Phase:** 1
- **설명:** CDN에서 모델 파일을 청크 단위로 다운로드한다. HTTP Range 요청을 사용하여 중단 후 재개가 가능하다.
- **인수 조건:**
  1. 다운로드 중 네트워크 단절 후 재시도 시 이어받기가 동작한다
  2. 동시 다운로드: 최대 2개 모델
  3. 다운로드 속도를 UI에 실시간 표시한다 (MB/s)
  4. 디스크 공간 부족 시 다운로드 전 경고한다
- **의존성:** FR-SRV-005

#### FR-MDL-003: 다운로드 진행률 표시

- **우선순위:** P0 | **Phase:** 1
- **설명:** 모델 다운로드 진행률을 프로그레스 바, 퍼센트, 예상 남은 시간으로 UI에 표시한다.
- **인수 조건:**
  1. 진행률이 0~100%로 표시된다
  2. 예상 남은 시간(ETA)이 표시된다
  3. 다운로드 취소 버튼이 제공된다
  4. 백그라운드 다운로드 시 시스템 알림으로 완료를 알린다
- **의존성:** FR-MDL-002

#### FR-MDL-004: 모델 무결성 검증

- **우선순위:** P0 | **Phase:** 1
- **설명:** 다운로드 완료 후 SHA-256 해시를 검증하여 파일 무결성을 확인한다.
- **인수 조건:**
  1. 해시 불일치 시 다운로드를 재시도하고 사용자에게 알린다
  2. 검증 시간이 1GB 파일 기준 5초 이내이다
  3. 기존 모델 파일의 무결성도 수동 재검증할 수 있다
- **의존성:** FR-MDL-002

#### FR-MDL-005: 시스템 사양 기반 모델 추천

- **우선순위:** P1 | **Phase:** 2
- **설명:** 사용자 시스템의 가용 RAM, GPU VRAM, CPU 코어 수를 감지하여 최적의 모델 조합을 추천한다.
- **인수 조건:**
  1. 최초 실행 시 자동 하드웨어 감지 결과를 표시한다
  2. RAM 8GB: tiny/base STT + 3B LLM 추천
  3. RAM 16GB + GPU: large-v3-turbo STT + 7B LLM 추천
  4. RAM 32GB + GPU: large-v3-turbo STT + 14B LLM 추천
  5. 추천과 무관하게 사용자가 자유롭게 모델을 선택할 수 있다
- **의존성:** FR-MDL-001

#### FR-MDL-006: 모델 핫스왑

- **우선순위:** P1 | **Phase:** 2
- **설명:** 앱 재시작 없이 런타임에 모델을 교체한다. 기존 모델 언로드 → 메모리 해제 → 새 모델 로드 과정을 수행한다.
- **인수 조건:**
  1. 모델 교체 시 진행 중인 작업(녹음, 전사)이 중단되지 않는다
  2. 현재 작업 완료 후 다음 요청부터 새 모델이 적용된다
  3. 메모리 해제가 확인된 후 새 모델을 로드한다 (메모리 이중 점유 방지)
  4. 교체 소요 시간: 7B 모델 기준 5초 이내
- **의존성:** FR-LLM-001, FR-STT-001

#### FR-MDL-007: 커스텀 GGUF 모델 로드

- **우선순위:** P2 | **Phase:** 2
- **설명:** 카탈로그에 없는 사용자 소유 GGUF 모델 파일을 직접 로드하여 사용할 수 있다.
- **인수 조건:**
  1. 파일 선택 다이얼로그 또는 경로 입력으로 GGUF 파일을 지정할 수 있다
  2. 모델 메타데이터(타입, 양자화, 컨텍스트 길이)를 자동 감지한다
  3. 호환되지 않는 모델 파일 로드 시 명확한 에러 메시지를 표시한다
- **의존성:** FR-LLM-001

### 3.11 FR-SRV: 서버 기능

서버는 최소 구성으로, 사용자 데이터를 일절 처리하지 않으며 인증, 라이선스, 알림, 동기화 릴레이, 모델 CDN만 담당한다.

---

#### FR-SRV-001: OAuth2 OIDC 로그인

- **우선순위:** P1 | **Phase:** 3
- **설명:** Google, Apple OAuth2 OIDC를 통한 소셜 로그인을 지원한다.
- **인수 조건:**
  1. Google 계정으로 로그인/회원가입이 가능하다
  2. Apple 계정으로 로그인/회원가입이 가능하다 (iOS App Store 요구사항)
  3. 로그인 없이도 로컬 전용 모드로 앱 사용이 가능하다
  4. 로그인 시 JWT(RS256) 토큰이 발급된다
- **의존성:** 없음

#### FR-SRV-002: JWT 토큰 관리

- **우선순위:** P1 | **Phase:** 3
- **설명:** JWT(RS256) 기반 인증 토큰 발급, 갱신, 무효화를 관리한다.
- **인수 조건:**
  1. Access Token 유효기간: 1시간
  2. Refresh Token 유효기간: 30일
  3. 토큰 갱신이 만료 전 자동으로 수행된다 (클라이언트)
  4. 로그아웃 시 서버에서 Refresh Token이 무효화된다
- **의존성:** FR-SRV-001

#### FR-SRV-003: 라이선스 관리

- **우선순위:** P1 | **Phase:** 3
- **설명:** 라이선스 키 검증 및 디바이스 활성화/비활성화를 관리한다.
- **인수 조건:**
  1. 라이선스 키로 디바이스를 활성화할 수 있다
  2. 라이선스당 최대 활성 디바이스 수를 설정할 수 있다 (기본: 5대)
  3. 사용자가 직접 디바이스를 비활성화하여 다른 디바이스에 양도할 수 있다
  4. 오프라인 시 캐시된 라이선스 상태로 72시간 동안 동작한다
- **의존성:** FR-SRV-001

#### FR-SRV-004: 푸시 알림

- **우선순위:** P2 | **Phase:** 4
- **설명:** FCM(Android/Web), APNs(iOS/macOS)를 통한 푸시 알림을 지원한다.
- **인수 조건:**
  1. 동기화 완료, 팀 공유 노트 업데이트 시 푸시 알림이 발송된다
  2. 사용자가 알림 카테고리별 수신 여부를 설정할 수 있다
  3. 알림 토큰 등록/갱신이 자동으로 수행된다
- **의존성:** FR-SRV-001

#### FR-SRV-005: 모델 CDN 배포

- **우선순위:** P0 | **Phase:** 1
- **설명:** Cloudflare R2/S3에서 AI 모델 파일을 CDN으로 배포한다. Signed URL로 인증된 다운로드를 제공한다.
- **인수 조건:**
  1. 모델 카탈로그 API가 사용 가능한 모델 목록을 반환한다
  2. 다운로드 URL은 Signed URL(TTL 1시간)로 제공된다
  3. CDN 엣지 캐싱으로 전세계 다운로드 속도를 보장한다
  4. 모델 파일 업데이트 시 캐시 무효화가 수행된다
- **의존성:** 없음

#### FR-SRV-006: CRDT 동기화 릴레이

- **우선순위:** P1 | **Phase:** 3
- **설명:** WebSocket으로 암호화된 CRDT 델타를 수신하여 해당 계정의 다른 디바이스들에게 중계한다.
- **인수 조건:**
  1. WebSocket 연결 관리: 계정당 최대 10개 동시 연결
  2. 델타 중계 지연: 100ms 이내 (서버 처리 시간)
  3. 오프라인 디바이스용 버퍼링: TTL 30일, 디바이스당 100MB
  4. 릴레이 서버의 수평 확장이 가능하다 (Redis pub/sub 기반)
- **의존성:** FR-SRV-001

#### FR-SRV-007: Rate Limiting

- **우선순위:** P1 | **Phase:** 3
- **설명:** 서버 API에 Rate Limiting을 적용하여 남용을 방지한다.
- **인수 조건:**
  1. 인증 API: 분당 10회
  2. 동기화 API: 분당 60회
  3. 모델 다운로드: 시간당 10회
  4. 한도 초과 시 429 상태 코드와 Retry-After 헤더를 반환한다
- **의존성:** 없음

#### FR-SRV-008: 사용자 프로필 관리

- **우선순위:** P1 | **Phase:** 3
- **설명:** 사용자 프로필 조회, 수정, 계정 삭제를 지원한다.
- **인수 조건:**
  1. 프로필 필드: 닉네임, 프로필 이미지, 이메일 (OIDC에서 획득)
  2. 계정 삭제 시 서버의 모든 사용자 데이터가 30일 이내에 완전 삭제된다
  3. 계정 삭제 전 확인 절차(이메일 인증)를 거친다
- **의존성:** FR-SRV-001

### 3.13 FR-MOB: 모바일 전략

모바일 디바이스의 리소스 제약을 고려한 계층형 전략을 적용한다.

---

#### FR-MOB-001: Tier 1 — 온디바이스 경량 모델

- **우선순위:** P1 | **Phase:** 3
- **설명:** 모바일에서 Whisper tiny/base(40~80MB)와 Qwen2.5-1.5B Q4(1GB)로 기본 전사·요약 기능을 제공한다.
- **인수 조건:**
  1. Whisper tiny/base 모델로 실시간 전사가 동작한다 (RTF ≥ 1.0)
  2. Qwen2.5-1.5B로 기본 요약이 생성된다
  3. 배터리 소모: 1시간 녹음 시 배터리 20% 이내 소모
  4. 앱 크기: 기본 모델 포함 시 200MB 이내
- **의존성:** FR-STT-001, FR-LLM-001

#### FR-MOB-002: Tier 2 — LAN 데스크탑 위임

- **우선순위:** P2 | **Phase:** 3
- **설명:** 같은 LAN의 VoxNote 데스크탑을 mDNS로 자동 탐색하고, E2EE 터널을 통해 대형 모델 추론을 위임한다.
- **인수 조건:**
  1. mDNS(_voxnote._tcp)로 LAN 내 데스크탑을 자동 발견한다
  2. 연결 시 E2EE 터널(TLS + X25519)을 수립한다
  3. 오디오 데이터는 암호화되어 전송되며, 데스크탑 측에서만 복호화된다
  4. 데스크탑 연결 끊김 시 자동으로 Tier 1(온디바이스)로 폴백한다
- **의존성:** FR-MOB-001

#### FR-MOB-003: Tier 3 — 클라우드 API (opt-in)

- **우선순위:** P1 | **Phase:** 3
- **설명:** 사용자가 직접 설정한 외부 API 엔드포인트(Ollama, 클라우드 LLM 등)를 사용한다.
- **인수 조건:**
  1. FR-API의 모든 Provider가 모바일에서도 동일하게 사용 가능하다
  2. 데이터 전송 시 사용자 동의를 요구한다 (최초 설정 시)
  3. Wi-Fi 전용 모드를 설정할 수 있다 (셀룰러 데이터 사용 방지)
- **의존성:** FR-API-001

#### FR-MOB-004: 모바일 클라우드 API 우선 옵션

- **우선순위:** P1 | **Phase:** 3
- **설명:** 모바일에서 디바이스 리소스 절약을 위해 외부 LLM Provider API를 기본 엔진으로 설정하는 옵션을 제공한다.
- **인수 조건:**
  1. 설정에서 "클라우드 우선 모드"를 활성화할 수 있다
  2. 활성화 시 LLM/STT가 외부 API로 기본 설정된다
  3. 네트워크 불가 시 자동으로 온디바이스 모델로 폴백한다
  4. 데이터 사용량 경고를 표시한다
- **의존성:** FR-API-001, FR-API-010

#### FR-MOB-005: Tauri Mobile 네이티브 앱

- **우선순위:** P1 | **Phase:** 3
- **설명:** Tauri 2.0 Mobile을 사용하여 iOS/Android 네이티브 앱을 빌드한다.
- **인수 조건:**
  1. iOS 16+, Android 12+ 에서 정상 동작한다
  2. 네이티브 UI 요소(상태바, 노치, 제스처)와 적절히 통합된다
  3. App Store / Google Play 심사 가이드라인을 준수한다
  4. 딥링크(voxnote://)를 지원한다
- **의존성:** FR-AUD-008, FR-AUD-009

-- VoxNote SQLite Schema v1.1

-- 폴더
CREATE TABLE folders (

);

CREATE TABLE notes (
    id          TEXT PRIMARY KEY,
    title       TEXT NOT NULL,
    folder_id   TEXT REFERENCES folders(id),
    status      TEXT NOT NULL DEFAULT 'draft',    -- 'draft' | 'recording' | 'completed'
    duration_ms INTEGER,                           -- 녹음 길이
    language    TEXT DEFAULT 'auto',
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_notes_folder ON notes(folder_id);
CREATE INDEX idx_notes_updated ON notes(updated_at DESC);

-- 전사 텍스트
CREATE TABLE transcripts (
    id            TEXT PRIMARY KEY,
    note_id       TEXT NOT NULL REFERENCES notes(id) ON DELETE CASCADE,
    timestamp_ms  INTEGER NOT NULL,
    end_ms        INTEGER NOT NULL,
    text          TEXT NOT NULL,
    speaker_id    TEXT,
    confidence    REAL,
    is_corrected  INTEGER DEFAULT 0,               -- 오탈자 교정 여부
    original_text TEXT                               -- 교정 전 원본 (교정 시에만)
);
CREATE INDEX idx_transcripts_note ON transcripts(note_id, timestamp_ms);

-- 요약
CREATE TABLE summaries (
    id          TEXT PRIMARY KEY,
    note_id     TEXT NOT NULL REFERENCES notes(id) ON DELETE CASCADE,
    template_id TEXT,
    content     TEXT NOT NULL,
    model_used  TEXT NOT NULL,                     -- 'local:qwen25-7b' | 'openai:gpt-4o' 등
    provider    TEXT NOT NULL DEFAULT 'local',     -- 'local' | 'openai' | 'anthropic' | ...
    version     INTEGER DEFAULT 1,                 -- 요약 재생성 시 버전 증가
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_summaries_note ON summaries(note_id);

-- 사용자 단어장
CREATE TABLE vocabulary (
    id          TEXT PRIMARY KEY,
    term        TEXT NOT NULL,
    replacement TEXT NOT NULL,
    domain      TEXT,                              -- 'general' | 'medical' | 'legal' | ...
    is_active   INTEGER DEFAULT 1,
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_vocabulary_domain ON vocabulary(domain);

-- 임베딩 (RAG용)
CREATE TABLE embeddings (
    id          TEXT PRIMARY KEY,
    note_id     TEXT NOT NULL REFERENCES notes(id) ON DELETE CASCADE,
    chunk_idx   INTEGER NOT NULL,
    chunk_text  TEXT NOT NULL,                     -- 원본 텍스트 청크
    vector      BLOB NOT NULL,                     -- f32 배열 직렬화
    model_id    TEXT NOT NULL,                     -- 임베딩 모델 식별
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_embeddings_note ON embeddings(note_id, chunk_idx);

-- 화자 프로필
CREATE TABLE speaker_profiles (
    id          TEXT PRIMARY KEY,
    name        TEXT NOT NULL,                     -- 사용자 할당 이름
    embedding   BLOB NOT NULL,                     -- 대표 임베딩 벡터
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

-- 템플릿
CREATE TABLE templates (
    id          TEXT PRIMARY KEY,
    name        TEXT NOT NULL,
    content     TEXT NOT NULL,                     -- Markdown 템플릿 본문
    is_builtin  INTEGER DEFAULT 0,                 -- 기본 제공 여부
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Provider 설정
CREATE TABLE provider_config (
    id          TEXT PRIMARY KEY,
    engine_type TEXT NOT NULL,                     -- 'stt' | 'llm' | 'tts'
    provider    TEXT NOT NULL,                     -- 'local' | 'openai' | 'anthropic' | ...
    model_id    TEXT NOT NULL,
    endpoint    TEXT,                              -- 사용자 정의 엔드포인트 (nullable)
    is_active   INTEGER DEFAULT 1,
    config_json TEXT,                              -- Provider별 추가 설정 (JSON)
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE UNIQUE INDEX idx_provider_engine ON provider_config(engine_type) WHERE is_active = 1;

-- API 사용량 로그
CREATE TABLE api_usage_log (
    id              TEXT PRIMARY KEY,
    provider        TEXT NOT NULL,
    model_id        TEXT NOT NULL,
    engine_type     TEXT NOT NULL,
    input_tokens    INTEGER,
    output_tokens   INTEGER,
    cost_usd        REAL,                          -- 예상 비용 (USD)
    created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_usage_provider_date ON api_usage_log(provider, created_at);

-- 전문검색 인덱스
CREATE VIRTUAL TABLE transcript_fts USING fts5(
    text,
    content='transcripts',
    content_rowid='rowid',
    tokenize='unicode61'
);

-- FTS 자동 동기화 트리거
CREATE TRIGGER transcript_ai AFTER INSERT ON transcripts BEGIN
    INSERT INTO transcript_fts(rowid, text) VALUES (new.rowid, new.text);
END;
CREATE TRIGGER transcript_ad AFTER DELETE ON transcripts BEGIN
    INSERT INTO transcript_fts(transcript_fts, rowid, text) VALUES ('delete', old.rowid, old.text);
END;
CREATE TRIGGER transcript_au AFTER UPDATE ON transcripts BEGIN
    INSERT INTO transcript_fts(transcript_fts, rowid, text) VALUES ('delete', old.rowid, old.text);
    INSERT INTO transcript_fts(rowid, text) VALUES (new.rowid, new.text);
END;

-- 스키마 버전 관리
CREATE TABLE schema_version (
    version     INTEGER PRIMARY KEY,
    applied_at  TEXT NOT NULL DEFAULT (datetime('now')),
    description TEXT
);
INSERT INTO schema_version (version, description) VALUES (1, 'Initial schema v1.1');
